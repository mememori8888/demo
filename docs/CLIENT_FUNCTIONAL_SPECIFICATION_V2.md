# Google Maps データ収集・分析アプリケーション 機能詳細仕様書

## 1. はじめに

本ドキュメントは、Google Maps上の施設情報を収集、更新、分析し、外部システムと連携するためのアプリケーションの全機能詳細を記述したものです。
GitHub Actionsのワークフロー (`issue-ops-universal.yml`) を中心に、各Pythonスクリプトが実行する具体的な処理ロジック、特に営業ステータスの更新やデータの正規化などの詳細仕様を網羅しています。

## 2. システム全体像

本システムは、**Issue Ops** と呼ばれる手法を採用しており、GitHub Issues上のコマンド操作によってデータ収集ジョブを制御します。

*   **インターフェース**: GitHub Issues (コマンド入力、進捗確認、結果レポート)
*   **オーケストレーション**: GitHub Actions (`issue-ops-universal.yml`)
*   **処理エンジン**: Python 3.11
*   **データソース**: Bright Data SERP API (Google Maps)
*   **データストレージ**: CSVファイル (Gitリポジトリ内で管理)

## 3. 主要機能（ワークフロー別詳細）

システムは以下の4つの主要なワークフローで構成されています。

### 3.1 レビュー取得機能 (`/run-reviews`)

指定された施設のGoogle Mapsレビュー情報を収集します。

*   **実行コマンド**: `/run-reviews`
*   **主要スクリプト**: `reviews_BrightData_50.py`
*   **入力データ**:
    *   FIDリスト (CSVファイル): 収集対象の施設ID (FID) が記載されたファイル。
    *   設定ファイル (`settings.json`): ターゲット設定など。
*   **出力データ**:
    *   レビューデータCSV: レビュワー名、評価、本文、投稿日時など。
*   **詳細仕様**:
    *   **FIDベースの検索**: 施設名や住所ではなく、Google Mapsの内部IDである **FID (Feature ID)** を使用してピンポイントでレビューを取得します。これにより、同名他店などの誤取得を防ぎます。
    *   **重複排除**: `レビューGID` (Google ID) をキーとして、既に取得済みのレビューはスキップし、新規レビューのみを追加保存します。
    *   **並列処理**: `ThreadPoolExecutor` を使用し、複数の施設に対するリクエストを並列実行して高速化しています。

### 3.2 施設情報取得機能 (`/run-facility`)

住所リストに基づいてGoogle Mapsを検索し、詳細な施設情報を収集します。営業ステータスの更新もこの機能に含まれます。

*   **実行コマンド**: `/run-facility`
*   **主要スクリプト**: `facility_BrightData_20.py`
*   **入力データ**:
    *   住所リスト (CSV): 検索対象の住所一覧。
*   **出力データ**:
    *   施設情報CSV: 施設名、電話番号、住所、営業ステータスなど。
    *   FIDファイル: 取得した施設のFIDリスト。
    *   重複分析レポート: 新規・既存・除外の判定結果。
*   **詳細仕様**:
    *   **検索クエリ構築**: 入力された住所にベースクエリ（例：「歯科医院」）を組み合わせて検索を実行します。
    *   **データ抽出項目**:
        *   基本情報: 施設名、電話番号、Webサイト、Google Map URL
        *   位置情報: 緯度、経度
        *   評価情報: ランク（星の数）、カテゴリ
        *   **ID情報**: 施設GID (Google ID), 施設FID (Feature ID)
    *   **営業ステータス更新ロジック**:
        APIから取得した `business_status` および関連フィールドを以下のルールで日本語ステータスに変換します。
        *   `OPERATIONAL` → **「営業中」**
        *   `CLOSED_TEMPORARILY` → **「一時休業」**
        *   `CLOSED_PERMANENTLY` → **「閉業」**
        *   上記以外で `permanently_closed` フラグが `true` の場合 → **「閉業」**
        *   上記以外で `opening_hours.open_now` が `true` の場合 → **「営業中」**
    *   **住所正規化ロジック**:
        取得した住所文字列を以下の手順で構造化データに変換します。
        1.  **郵便番号・都道府県抽出**: 正規表現を用いて `〒xxx-xxxx` および都道府県名を抽出。
        2.  **市区町村分離**: 都道府県を除去した後、正規表現 `^([^市区町村]+(?:市|区|郡|町|村))` を用いて市区町村部分を特定。
        3.  **郡・町村の扱い**: 「郡」で終わる場合は、続く「町」や「村」までを市区町村として結合（例：「北松浦郡佐々町」）。
        4.  **ゼロ落ち防止**: 郵便番号や電話番号の先頭にゼロ幅スペース (`\u200b`) を付与し、Excel等で開いた際の「0落ち」を防止。
    *   **ID管理と重複排除**:
        *   **GID (Google ID)** をユニークキーとして使用。
        *   既存のCSVファイルに存在するGIDはスキップ（「既存」判定）。
        *   除外リストに含まれるGIDはスキップ（「除外GID」判定）。
        *   新規施設には、既存の最大ID + 1 から連番で **施設ID** を自動採番。

### 3.3 ヒートマップ用施設取得機能 (`/run-facility-heatmap`)

広範囲のエリア分析（ヒートマップ作成）のために、大量の施設情報を効率的に収集します。

*   **実行コマンド**: `/run-facility-heatmap`
*   **主要スクリプト**: `facility_BrightData_heatmap.py`
*   **入力データ**:
    *   ヒートマップ用検索リスト: エリアごとの検索クエリと推奨リクエスト回数が記載されたCSV。
*   **出力データ**:
    *   施設情報CSV (ヒートマップ用)
*   **詳細仕様**:
    *   **動的リクエスト制御**: エリアごとの施設密度に応じて、APIリクエスト回数 (`max_requests`) を動的に調整します。施設が多いエリアは複数ページ取得し、少ないエリアは1回で済ませることで、コストと網羅性のバランスを最適化しています。
    *   **バッチ処理**: 大量の検索対象を `batch_size` (デフォルト2000件) ごとに分割して処理可能。GitHub Actionsのタイムアウトを回避しながら、数万件のデータを確実に処理します。
    *   **スナップショットAPI対応**: Bright Dataの新しい非同期API (Snapshot ID方式) に対応し、大規模データの取得安定性を向上させています。

### 3.4 ヒートマップ生成機能 (`/run-generate-heatmap`)

収集された施設データを元に、ヒートマップ作成に必要なデータ処理を行います。

*   **実行コマンド**: `/run-generate-heatmap`
*   **主要スクリプト**: `search_optimizer.py` (オプション `--heatmap-only`)
*   **機能**:
    *   収集された施設データを集計・加工し、可視化ツール（Google My Maps等）で読み込み可能な形式や、分析用の中間データを生成します。

## 4. 共通仕様・技術詳細

### 4.1 コスト管理・見積もり機能

ジョブ実行前に、リクエスト数と概算コストを自動計算して提示します。

*   **見積もりロジック**:
    *   `reviews`: 施設数 × 5 リクエスト
    *   `facility`: 住所数 × 1 リクエスト
    *   `facility_heatmap`: バッチサイズ × リクエスト単価
*   **承認フロー**: 見積もりコストを確認後、管理者が `/承認` コメントを投稿することで初めて処理が実行される安全設計です。

### 4.2 データ整合性管理

*   **排他制御**: 並列処理時のファイル書き込み競合を防ぐため、`threading.Lock` を使用してスレッドセーフな実装を行っています。
*   **途中保存**: 処理が長時間に及ぶ場合でもデータ消失を防ぐため、一定件数（例: 50件）ごとにデータをディスクにフラッシュ（保存）します。
*   **エラーハンドリング**:
    *   APIエラー（5xx, 429等）に対する指数バックオフ（Exponential Backoff）付きリトライ。
    *   JSONパースエラーやネットワークタイムアウトの適切な捕捉とログ記録。

### 4.3 Bright Data API連携

*   **ハイブリッドAPI利用**:
    *   **Web Unlocker API**: 従来の同期型API。即時性が求められる処理に使用。
    *   **Scraper API (New)**: 新しい非同期スナップショットAPI。大量データの取得や、より詳細なデータ構造が必要な場合に対応（`USE_NEW_API` フラグで切り替え可能）。
*   **検索パラメータ**:
    *   `hl=ja`: 日本語設定
    *   `gl=jp`: 地域設定（日本）
    *   `brd_json=1`: JSON形式でのレスポンス取得

### 4.4 入力データの処理と正規化

システムは、入力されたデータ（CSVファイル）に対して、実行時に自動的なクリーニングと正規化を行います。

#### 1. 住所リストの処理 (`/run-facility`)
*   **読み込み**: CSVファイルの各行を読み込み、スペース区切りで結合して検索クエリ化します。
*   **重複チェック**: 検索実行前に、既に取得済みの施設（GIDが一致するもの）や、除外リストに含まれる施設は自動的にスキップされます。

#### 2. FIDリストの処理 (`/run-reviews`)
*   **自動重複排除**: 入力されたFIDリスト（CSV）内に同一のFIDが複数含まれている場合、読み込み時に自動的に重複を削除し、ユニークなリストとして処理します。これにより、無駄なAPIリクエストの発生を防ぎます。
*   **列の自動検出**: CSVファイル内の列名（ヘッダー）を解析し、`施設FID`, `FID`, `fid` などの列を自動的に特定して読み込みます。

#### 3. Google Map URLからのFID抽出 (補助機能)
Google MapのURLリストしか手元にない場合でも、補助ツール (`extract_fid_from_urls.py`) を使用してFIDを抽出可能です。

*   **リダイレクト追跡**: 短縮URLやリダイレクトが含まれるURLでも、Seleniumを使用して最終的なURLまで追跡し、正確なFID (`0x...:0x...`) を抽出します。
*   **対応フォーマット**:
    *   通常のPlace URL (`/maps/place/...`)
    *   座標付きURL (`/maps/@...`)
    *   短縮URL形式

## 5. 運用フロー (Issue Ops)

1.  **Issue作成**: GitHubで新しいIssueを作成し、本文に対象のコマンド（例: `/run-facility`）とパラメータ（JSON形式）を記述します。
2.  **プレビュー**: システムが自動的にリクエスト内容を解析し、予想リクエスト数とコストをコメントとして返信します。
3.  **承認**: 内容に問題がなければ、管理者が `/承認` とコメントします。
4.  **実行**: 承認をトリガーとしてPythonスクリプトが実行され、データ収集が開始されます。
5.  **完了報告**: 処理が完了すると、結果ファイルのリンク、取得件数、ログへのリンクを含む完了レポートがコメントされ、Issueが自動的にクローズされます。
6.  **データ確認**: リポジトリの `results/` フォルダに保存されたCSVファイルを確認・ダウンロードします。
