# Sequential Job オーケストレーション設計ガイド

## 📊 現在のオーケストレーション構造

このワークフローは **完全カスタム実装** の複雑なシーケンシャル処理です。

### 1️⃣ 多層的な待機・リトライ機構

```
入力CSV
  ↓
[CSV_BATCH_SIZE単位で分割] → 500行ごと
  ↓
[API_BATCH_SIZE単位で分割] → 50件ごと
  ↓
BrightData API呼び出し
  ↓
スナップショット完成待機 (MAX_WAIT_MINUTES)
  ↓
データダウンロード
  ↓
Git自動保存＆Push
  ↓
バッチ間待機 (WAIT_TIME)
  ↓
次のバッチへ
```

---

## ⏱️ 待機時間設定の詳細

### ワークフロー内の設定値

```yaml
# 📄 .github/workflows/dental_new_reviews_sequential.yml

CSV_BATCH_SIZE=500           # CSVを500行ごとに分割（ファイル分割用）
WAIT_TIME=120                # バッチ間の待機時間（秒） = 2分
API_BATCH_SIZE=50            # API 1回あたりの処理件数
MAX_WAIT_MINUTES=90          # スナップショット待機の最大時間 = 90分
```

### Pythonスクリプト内の設定

```python
# 📄 get_reviews_from_dental_new.py

BATCH_SIZE = int(os.getenv('BATCH_SIZE', '100'))        # ← ワークフローから API_BATCH_SIZE に置換
MAX_WAIT_MINUTES = int(os.getenv('MAX_WAIT_MINUTES', '60'))  # スナップショット待機時間
```

## 🔄 処理フロー詳細

### ステップ1: CSVファイルの分割

```
例）1000行のCSVファイルを処理する場合
CSV_BATCH_SIZE = 500 の場合:

Batch 1: 行1～500 (500件)
Batch 2: 行501～1000 (500件)

→ 合計2バッチ
```

### ステップ2: 各バッチの内部処理

```
バッチ 1 (500件の施設)
  ├─ API_BATCH_SIZE = 50に分割
  │  ├─ Sub-batch 1: 施設1～50
  │  ├─ Sub-batch 2: 施設51～100
  │  ├─ ...
  │  └─ Sub-batch 10: 施設451～500
  │
  ├─ 各Sub-batchに対して
  │  ├─ Step 1: APIトリガー
  │  │  └─ スナップショットID取得
  │  │
  │  ├─ Step 2: スナップショット待機
  │  │  └─ 最大90分、15秒ごとにポーリング
  │  │     条件1: status = "ready" → 次へ進む
  │  │     条件2: status = "failed" → エラー
  │  │     条件3: 90分超過 → タイムアウト
  │  │
  │  └─ Step 3: データダウンロード
  │     └─ スナップショットからレビューを取得
  │
  └─ step別の処理後
     └─ 120秒待機（API制限対策）
        └─ 次のBatch 2へ
```

### ステップ3: バックグラウンド自動保存

```
メイン処理と並行して:

定期自動保存スレッド (30分ごと)
  ├─ 30分待機
  ├─ Git ロック取得
  ├─ 変更ファイルをコミット
  ├─ リモートにプッシュ
  └─ ロック解放
     → 次の30分へ

目的: GitHub Actions のタイムアウト（6時間）前に
      途中データを必ずpushしておく
```

## 📈 処理時間の計算方法

### 総処理時間の予測

```
総処理時間 = Σ(バッチ処理時間) + Σ(バッチ間待機時間)

1バッチの処理時間 = APIトリガー時間 + スナップショット待機時間 + データダウンロード時間

スナップショット待機時間は最も長く:
  - 平均: 5～30分
  - 最長: MAX_WAIT_MINUTES = 90分
```

### 例）1000件の施設を処理する場合

```
設定:
  - CSV_BATCH_SIZE = 500 (2バッチ)
  - API_BATCH_SIZE = 50 (sub-batch単位)
  - 平均スナップショット待機時間 = 15分

予測:
  バッチ1: 500件 → APIトリガー～ダウンロード: 約15～20分
  バッチ間待機: 2分
  バッチ2: 500件 → APIトリガー～ダウンロード: 約15～20分
  ───────────────────────────
  総計: 約35～45分
```

## ⚡ タイミング設定の最適化ガイド

### シナリオ別の推奨設定

#### 📍 シナリオ1: 小規模データ（施設1～100件）

```yaml
推奨設定:
  CSV_BATCH_SIZE: 100       # 通常: 500
  API_BATCH_SIZE: 25        # 通常: 50
  WAIT_TIME: 60             # 通常: 120
  MAX_WAIT_MINUTES: 60      # 通常: 90

理由:
  - 小規模なので細分化不要
  - レスポンス待機時間も短い
  - 全体処理時間: 10～20分
```

#### 📍 シナリオ2: 中規模データ（施設100～500件）

```yaml
推奨設定:
  CSV_BATCH_SIZE: 500       # ← 推奨（デフォルト）
  API_BATCH_SIZE: 50        # ← 推奨（デフォルト）
  WAIT_TIME: 120            # ← 推奨（デフォルト）
  MAX_WAIT_MINUTES: 90      # ← 推奨（デフォルト）

理由:
  - 標準設定で問題なし
  - 全体処理時間: 30～60分
```

#### 📍 シナリオ3: 大規模データ（施設500件以上）

```yaml
推奨設定:
  CSV_BATCH_SIZE: 500
  API_BATCH_SIZE: 25        # 削減
  WAIT_TIME: 180            # 増加
  MAX_WAIT_MINUTES: 120     # 増加

理由:
  - API呼び出しを小さくしてタイムアウト回避
  - バッチ間の待機を増やしてAPI負荷軽減
  - 1回の処理は遅くなるが、信頼性向上
  - 全体処理時間: 60～120分
```

#### 📍 シナリオ4: 頻繁に実行する場合

```yaml
推奨設定:
  CSV_BATCH_SIZE: 250       # 縮小分割
  API_BATCH_SIZE: 30
  WAIT_TIME: 90
  MAX_WAIT_MINUTES: 60      # 短縮

理由:
  - 小分割により高速化
  - 定期実行（日1回など）の場合に最適
  - 全体処理時間: 20～40分
```

## 🚨 タイムアウト対策

### ❌ 避けるべき設定

```yaml
# ❌ NGな例
MAX_WAIT_MINUTES: 300       # 5時間待機 → 6時間制限を超過可能
WAIT_TIME: 600              # バッチ間10分待機 → 処理時間が長すぎる
API_BATCH_SIZE: 500         # 1回に500件 → タイムアウト確率UP
```

### ✅ 推奨される設定

```yaml
# ✅ 推奨設定
⏱️ GitHub Actions制限: 最大6時間
├─ 実際の有効時間: 345分（5時間45分）
│  └─ ワークフロー: timeout-minutes: 350
│     で345分ポイントで警告が出る
│
└─ 安全マージン: 最低15分確保
   └─ 危険ライン: MAX_WAIT_MINUTES × バッチ数 > 330分
```

## 📋 ログ出力を見て判断するポイント

### パターン1: 正常に動作している場合

```log
✅ ポーリング数が多い（10～50回）
✅ ステータスが: collecting → digesting → ready と進行
✅ 各バッチの処理時間が安定している（±5分以内）
✅ 新規レビュー数が増加している
```

### パターン2: 待機が長い場合

```log
⚠️ スナップショット待機が30分以上
⚠️ ステータスが collecting のまま変わらない
⚠️ ポーリング数が100回以上

対策:
  - API_BATCH_SIZE を削減
  - 入力データのサイズを確認
  - BrightData API の負荷状態を確認
```

### パターン3: APIエラーが頻発する場合

```log
❌ HTTP 429 (Rate Limited)
❌ HTTP 500 (Server Error)
❌ Connection timeout

対策:
  - WAIT_TIME を増やす（現在の120秒 → 300秒）
  - API_BATCH_SIZE を削減
  - ワークフロー実行の間隔を広げる
```

### パターン4: タイムアウト検出

```log
⏰ タイムアウト5分前に達しました
⏰ 次のバッチ後に強制終了します

この場合:
  - 途中データは自動保存されている
  - 再度実行する際は start_from_batch で続きから再開
  - MAX_WAIT_MINUTES を削減を検討
```

## 📊 実行例

### 実行例1: 100件の施設

```
設定: CSV_BATCH_SIZE=500, API_BATCH_SIZE=50, WAIT_TIME=120

処理:
  1バッチ: 100件
  APIチャンク: 100件を50件ずつ = 2チャンク
  
  チャンク1: APIトリガー(3秒) + 待機(12分) + ダウンロード(1秒) = 約12分
  チャンク2: APIトリガー(3秒) + 待機(8分) + ダウンロード(1秒) = 約8分
  
総処理時間: 12 + 8 = 約20分 ✅
```

### 実行例2: 1000件の施設

```
設定: CSV_BATCH_SIZE=500, API_BATCH_SIZE=50, WAIT_TIME=120

処理:
  バッチ1: 500件 = 約45分
  バッチ間待機: 2分
  バッチ2: 500件 = 約45分
  
総処理時間: 45 + 2 + 45 = 約92分 ✅
```

### 実行例3: 5000件の施設（大規模）

```
設定: CSV_BATCH_SIZE=500, API_BATCH_SIZE=50, WAIT_TIME=120

処理:
  バッチ1～10: 各約45分 = 450分
  バッチ間待機: 各2分 × 9 = 18分
  
総処理時間: 468分 ≈ 7.8時間 ⚠️ → GitHub制限(6時間)を超過！

対策: 分割実行
  - ワークフロー1: start_from_batch: 1 → 5
  - ワークフロー2: start_from_batch: 6 → 10
  - 各ワークフロー: 約240分（4時間）で完了 ✅
```
