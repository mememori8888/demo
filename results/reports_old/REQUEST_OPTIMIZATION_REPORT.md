# APIリクエスト最適化とコスト削減に関する報告書

## 1. 背景と課題

現在、検索対象となる住所リスト（`wordpress_only_heatmap_pref_firstnum.csv`）には **13,272件** の住所が含まれています。これら全てに対して個別にAPIリクエストを行うと、以下の問題が発生します。

1.  **膨大なコストと時間**: 13,000回以上のリクエストが発生し、API利用料と処理時間が増大する。
2.  **非効率な重複取得**: 近接する住所（例：「札幌市中央区南2条」と「南3条」）を個別に検索すると、同じ施設が何度も重複して取得される。
3.  **無駄なリクエスト**: 施設が少ないエリアに対しても過剰にリクエストを投げてしまう、あるいは逆に施設が多いエリアで取りこぼしが発生する可能性がある。

## 2. 実施した改善策

これらの課題を解決するため、**「地理的クラスタリング」** と **「実績データに基づくリクエスト数算出」** を組み合わせた最適化を行いました。

### ① 地理的クラスタリング（1km圏内の統合）

緯度経度情報を利用し、**半径1km以内** に位置する近接した住所グループを1つの検索ポイントに統合しました。

*   **Before**: 「札幌市中央区南2条」「札幌市中央区南3条」「札幌市中央区南4条」... と個別に検索。
*   **After**: これらを「札幌市中央区（中心座標）」として1つにまとめ、広範囲を一度に検索。

これにより、検索地点（ユニーク住所数）を大幅に削減しつつ、エリアを網羅することが可能になります。

### ② 施設数に基づくリクエスト回数の最適化

既存の取得データ（`dental.csv`）を分析し、各エリアに実際に存在する施設数を把握。その数に基づいて、必要なAPIリクエスト回数を自動算出するロジックを実装しました。

*   **計算式**: `推奨リクエスト回数 = 切り上げ( エリア内の施設数 / 20 )`
    *   ※Google Maps APIは1リクエストあたり最大20件取得できるため。

**例:**
*   **エリアA（施設数 100件）**: 100 / 20 = **5回** リクエスト（全件取得のため）
*   **エリアB（施設数 5件）**: 5 / 20 = **1回** リクエスト（無駄撃ちなし）

## 3. 改善の仕組み（イメージ）

| 項目 | 改善前（従来） | 改善後（今回） |
| :--- | :--- | :--- |
| **検索単位** | 住所文字列ごと（バラバラ） | **地理的エリアごと（1km圏内で統合）** |
| **リクエスト数** | 一律、または手動設定 | **施設数に応じて自動計算（最適化）** |
| **重複取得** | 多い（近隣住所で重複） | **最小限（エリアでまとめるため）** |
| **コスト** | 高（全住所分リクエスト） | **低（必要な分だけリクエスト）** |

## 4. 期待される効果

1.  **コストの大幅削減**:
    *   近接住所の統合により、検索地点数が減少します。
    *   施設数に合わせたピンポイントなリクエストにより、無駄なAPIコール（0件ヒットや過剰なページめくり）を排除します。

2.  **データ取得の効率化**:
    *   重複データの処理コストが下がります。
    *   必要なエリアには十分なリクエストを投げるため、取りこぼしを防ぎます。

3.  **柔軟な運用**:
    *   今後データが増えても、自動的に再計算を行い、常に最適なリクエスト計画を立てることができます。

## 5. 成果物

*   **最適化済みリスト**: `results/dental_heatmap_pref_firstnum.csv`
    *   統合された住所、中心座標、エリア内の施設数、推奨リクエスト回数が記載されています。
*   **生成ツール**: `create_dental_heatmap.py`
    *   最新のデータからいつでも最適化リストを再生成可能です。
*   **実行スクリプト**: `facility_BrightData_20_update.py`
    *   最適化リストを読み込み、自動的に効率的な収集を行います。
