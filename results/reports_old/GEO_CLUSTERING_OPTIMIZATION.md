# 地理的クラスタリングによるリクエスト削減

## 問題点

`wordpress_only_heatmap_pref_firstnum.csv` ファイルには13,272件の住所が含まれており、各住所に対して個別にAPIリクエストを送信すると、リクエスト数が膨大になってしまいます。

## 解決策

緯度経度を利用して地理的に近い住所をグループ化（クラスタリング）し、クラスター単位でまとめてリクエストすることで、リクエスト数を大幅に削減します。

## 仕組み

### 1. 住所から緯度経度を取得
- Nominatim（OpenStreetMap）のジオコーディングAPIを使用
- 取得した緯度経度はキャッシュして再利用

### 2. 地理的クラスタリング (DBSCAN)
- 緯度経度を使ってクラスタリング
- デフォルト設定: 半径15km以内、最小3件で1クラスター
- 近隣の住所をグループ化

### 3. クラスター単位で検索
- 各クラスターの代表住所（例: 「北海道札幌市」）で広域検索
- 施設数に応じてリクエスト回数を調整（20件/リクエスト）

### 4. 結果のフィルタリング
- 取得した施設を元の住所でフィルタリング
- 重複を排除

## 使用方法

### 1. 依存ライブラリのインストール

```bash
pip install -r requirements.txt
```

または個別に:

```bash
pip install geopy scikit-learn numpy
```

### 2. クラスタリングの実行

```bash
python facility_BrightData_20_update_geo.py
```

このスクリプトは以下を実行します:
1. `results/wordpress_only_heatmap_pref_firstnum.csv` から住所を読み込み
2. 各住所の緯度経度を取得（キャッシュあり）
3. 地理的クラスタリングを実行
4. `results/geo_clustered_addresses.csv` にクラスタ情報を出力
5. `settings/geo_search_queries.csv` に検索クエリを生成

### 3. 設定ファイルの更新

`settings/settings.json` を編集して、`address_csv_path` を変更:

```json
{
    "task_name": "dental_facility_search",
    "query": "歯医者",
    "address_csv_path": "geo_search_queries.csv",
    ...
}
```

### 4. 施設データの取得

通常通り実行:

```bash
python facility_BrightData_20_update.py
```

## パラメータ調整

`facility_BrightData_20_update_geo.py` の main() 関数内で調整可能:

```python
EPS_KM = 15          # クラスタ半径（km）
                     # - 小さくする: より細かいクラスタ、リクエスト増
                     # - 大きくする: より広範囲のクラスタ、リクエスト減

MIN_SAMPLES = 3      # 最小サンプル数
                     # - 小さくする: より多くのクラスタ
                     # - 大きくする: より大きなクラスタのみ作成

BASE_QUERY = "歯医者"  # 検索する業種
```

## 予想される効果

### 元の方法
- 13,272件の住所 → 13,272リクエスト

### 改善後の方法（推定）
- クラスタ数: 約500-1,000件
- リクエスト削減率: **90-95%**
- 13,272件 → 500-1,000リクエスト

実際の削減率はクラスタリングパラメータと地理的分布によって変わります。

## 出力ファイル

### `results/geo_clustered_addresses.csv`
クラスタリング結果:
- cluster_id: クラスターID（-1はノイズポイント）
- address_count: クラスター内の住所数
- center_lat, center_lon: クラスターの中心座標
- representative_address: 代表住所

### `settings/geo_search_queries.csv`
検索クエリ:
- cluster_id: クラスターID
- query: 検索クエリ（例: "北海道 札幌市"）
- base_query: 業種（例: "歯医者"）
- request_count: このクエリで必要なリクエスト回数
- facility_count: 期待される施設数
- lat, lon: 検索中心座標

### `results/geocode_cache.pkl`
ジオコーディング結果のキャッシュ（再実行時に高速化）

## 注意点

### 1. ジオコーディングの制限
- Nominatimは1秒に1リクエストの制限があります
- 初回実行は時間がかかります（キャッシュされるため2回目以降は高速）
- 大量の住所がある場合は数時間かかる可能性があります

### 2. 検索精度
- 広域検索のため、元の住所から離れた施設も取得される可能性があります
- フィルタリング処理で除外されますが、取得データ量は増えます

### 3. クラスタリングパラメータ
- 地域の密度によって最適なパラメータは異なります
- 都市部: 小さい半径（5-10km）
- 地方部: 大きい半径（15-30km）

## トラブルシューティング

### ジオコーディングが遅い
- キャッシュファイル `results/geocode_cache.pkl` を確認
- 既に取得済みの住所は再取得されません

### クラスタが少なすぎる/多すぎる
- `EPS_KM` と `MIN_SAMPLES` を調整してください
- 統計情報を見て適切な値を決定

### メモリエラー
- 住所数が非常に多い場合、バッチ処理に分割してください
- `create_geo_clustered_address_file()` 関数でバッチサイズを制限

## 今後の改善案

1. **多段階クラスタリング**
   - 都道府県 → 市区町村 → 詳細住所の階層的クラスタリング

2. **並列ジオコーディング**
   - 複数のジオコーディングサービスを並列利用

3. **スマートフィルタリング**
   - 取得した施設の住所を元の住所リストと照合してマッチング精度向上

4. **動的パラメータ調整**
   - 地域の密度に応じて自動的にクラスタ半径を調整
